Notes on releasing new versions of Crux

Note: A detailed checklist is at the bottom of this page

Before you begin

Confirm that a clean check out builds correctly and passes all the smoke and unit tests. Update the release notes (crux/doc/user/release-notes.html). At the very least, the date will need to be added. Update index.html in doc/user/downloads to point to the new version and to include links to the previous version. You can also make sure that the default.params file is up-to-date by running write-default-params.sh in crux/src/c.

Branching

The first release of crux, version 1.0, was given its own development branch named rel-1-0 in the SVN repository. Bugs reported by users were fixed in this branch and builds from this branch were made available for download. Changes made to this branch are periodically merged into the trunk. The next release should be given its own branch at which point work in rel-1-0 will cease and that directory can be deleted.

Version numbers

Each time new files are made available for download, the version number should increment by 0.01 for minor releases or by 0.1 for major releases. The version number is maintained in CMakeLists.txt in the root Crux directory in the lines 

set (CPACK_PACKAGE_VERSION_MAJOR "X")
set (CPACK_PACKAGE_VERSION_MINOR "XX")
This controls the numbering of packages and what is reported by the --version option.

Building packages

Avoid building packages in the root. Create a sub-directory in the root for the relase type (source, 32-bit Linux, 64-bit Linux, Darwin). Move into the sub-directory and run 

cmake ..

There are two targets for building packages. The command make package_source will create a tarred and zipped file containing source code and documentation. The command make package will create a zipped file containing the compiled binaries for all of the crux tools and the documentation. The binary package file will include the computer architecture and operating system in the name so that builds on multiple computers can be easily distinguished.

After building the packages, run make checksum. Do this for each bdist package built. A new file with the output of the checksum is created for each package and will be included on the download page.

The distribution files are no longer being stored in SVN. However, putting the files in crux/doc/user/downloads will facilitate the website update step. Detailed notes about changes made can be recorded in crux/ChangeLog. A more general description of changes (aimed at the user) should be recorded in crux/doc/user/download/release-notes.html

Updating the website

The public crux website is at /var/noble/www/htdocs/proj/crux and its address is http://noble.gs.washington.edu/proj/crux/. The proj/crux/ directory should contain all of the files and sub-directories of crux/doc/user/. The the target webpage in crux/doc/user will copy all necessary files to the web directory. The login/password for the downloads page is crux and warm.Brandy.

Checklist

Here is a list of steps to follow, roughly in this order

  1. Create a release branch.
  2. Do a clean checkout of the release branch.

    Update documentation

  3. Change version number in crux/CMakeLists.txt.
  4. Look at the SVN log since the last release and update release notes accordingly.
  5. Change the crux/doc/user/download/index.html page.

    Build and test

  6. Build.
  7. Check unit and smoke tests.
  8. Check version number with crux version.
  9. Commit changes.

    Create files for distribution

  10. Create packages
    • cmake ..
    • make package_source
    • make package
    • make checksum
    You should now have these files:
    • crux_1.xx.tar.gz
    • crux_1.xx.tar.gz.md5sum.txt
    • crux_1.xx-PLATFORM.zip
    • crux_1.xx-PLATFORM.zip.md5sum.txt
  11. Repeat the build and package steps for remaining platforms. Note that you do not need to package up the source files again.

    Check the distribution files

  12. Unpack and build the source file tarballs.
  13. Unpack and run the binary files. You may want to ask someone else to do these steps on different computers.

    Update the website

  14. Move the distribution files to crux/doc/user/download and crux/doc/user/open_source_download.
  15. Copy the documentation and distribution files to the website with make webpage from crux/doc/user.
  16. Test that the links all work.
  17. Send a message to the crux users list announcing the new release.
